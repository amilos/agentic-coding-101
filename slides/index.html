Welcome. This is a train-the-trainer session: you will both learn the material and learn to deliver it. Two hours, hands-on, built around a fictional retail bank "NorthBank" core. Framing: how we got here

1. Autocomplete

Line-by-line suggestions. You drive every keystroke.

2. Chat

Ask/edit in a side panel. You still copy, paste, run.

3. Agent-native

Delegate a goal. The agent plans, edits many files, runs tools, opens PRs.

This course lives in the third era: agent-native development.

Scope  We are using Copilot, not building agents.

Draw the arc: autocomplete (Copilot v1) → chat (side-panel Q&A) → agent-native (delegate whole tasks). The mental shift for a bank engineer: you move from author to reviewer/director. Emphasise scope repeatedly today: we are consumers of an agentic tool, not authors of a custom agent framework. That keeps the two hours focused.

What is the GitHub Copilot app?

  • The agent-native desktop app — Windows / macOS / Linux.
  • GA 17 June 2026.
  • Requires a paid Copilot licence.
  • NOT the VS Code plugin — it is repo/worktree-based.
  • Runs local sessions and GitHub-hosted cloud sessions.
Nail the facts: GA 17 June 2026, desktop, paid licence, all three OSes. Contrast with the VS Code Copilot plugin most people know: this is a standalone app organised around git worktrees, not an editor extension. It can run work locally or hand it to the GitHub-hosted cloud coding agent. This distinction matters for the Delphi station later.

Concept: sessions & worktrees

Every agent session runs in its own isolated git worktree — parallel work, no stepping on each other.

The worktree is the key safety property: an agent can churn on session B while you keep working, and nothing leaks between them. Show this live in the tour. Analogy: each session gets its own sandboxed copy of the working tree.

Concept: canvases

Shared surfaces that show the agent's plan, terminals, diffs and workflow state as it works.

Canvases are where the agent's thinking becomes visible and reviewable. As a trainer, keep pointing at the canvas during demos so people learn to read the plan and diff before they approve anything.

Concept: My Work

A single queue of your sessions, issues and PRs — your command centre.

My Work is where you triage. When cloud sessions finish, they surface here. Encourage trainers to start demos from My Work so attendees build the habit of a single queue rather than losing track of parallel agents.

Concept: Automations

Recurring / background prompts — the agent works on a schedule, not just on demand.

Automations turn Copilot into a background teammate: nightly dependency/CVE sweeps, compliance checks, doc refreshes. We wire one up in Station 7. Caution attendees on cost/noise: schedule deliberately.

Concept: Agent Merge

Guides a PR through review → checks → merge — with you approving the gates.

Agent Merge shepherds a PR the last mile. It is assistive, not autonomous: you still approve. In a bank, human review gates stay mandatory — make that explicit here and again on the guardrails slide.

Extending Copilot: Skills

Skills = folders of instructions/scripts Copilot loads when relevant.

  • A SKILL.md with YAML frontmatter (name, description).
  • Loaded on demand when the task matches.
  • Community: Addy Osmani agent-skills (24 lifecycle skills), Matt Pocock short-form skills, github/awesome-copilot.
  • Add a pack: copy skill folders into .github/skills/ — no marketplace needed.
.github/skills/banking-review/SKILL.md
---
name: banking-review
description: Review a diff for NorthBank money-safety rules
---
In the tour (0:10–0:25) we add Addy Osmani's agent-skills together: clone the pack and copy a couple of skill folders into the repo's .github/skills/ (it is not a Copilot marketplace, so there is no /plugin install for it). Skills are the lightest customisation: a markdown file with frontmatter. Attendees author their own in Station 6.

Extending Copilot: Plugins

Plugins bundle custom agents, skills, hooks, MCP servers and LSP configs.

  • Custom agents: *.agent.md
  • Hooks: hooks.json · MCP: .mcp.json / mcp.json
  • Install: /plugin install owner/repo (Copilot CLI; repo must be a marketplace)
Plugins are how a bank platform team packages house rules and ships them to everyone. Note the mechanism: /plugin install is a Copilot-CLI command and only resolves repos that publish a marketplace.json; the app enables plugins via .github/copilot/settings.json. The demo-repo ships a sample security-reviewer.agent.md to show the shape of a custom agent.

Extending Copilot: MCP

Model Context Protocol servers wire external tools/data into repo and coding-agent settings.

MCP is the standard socket for giving the agent new capabilities and data sources. The repo ships a template mcp.json with a filesystem server plus a commented GitHub server. Keep the security lens: every MCP server is new surface area.

Advanced · 1:25–1:45

Spec-driven development

OpenSpec: propose → apply → archive.

After the break. This is the "level up" segment — specs before code, tracked as artefacts.

OpenSpec · propose → apply → archive

Lightweight, brownfield-first spec-driven development. Generates .github/prompts/*.prompt.md that the Copilot app surfaces as slash commands.

openspec propose add-transfer-limit
# review generated proposal/specs/tasks
openspec apply  add-transfer-limit
openspec archive add-transfer-limit

Contrast: GitHub Spec Kit

Heavier — templates + CLI, /specify /plan /tasks, agent-agnostic (30+ agents). Choose OpenSpec for brownfield & light-touch; Spec Kit for greenfield & ceremony.

OpenSpec (Fission-AI) is lightweight and brownfield-first; the propose/apply/archive verbs appear as slash-command prompt files. Spec Kit (github/spec-kit) is the heavier, template-driven, agent-agnostic alternative. Position them as two points on a spectrum, not competitors. Reuse add-transfer-limit so it mirrors the station work.

Guardrails for a regulated bank

  • Human review is mandatory — Agent Merge assists, it does not replace approval.
  • Read the diff and plan on the canvas before applying.
  • Money is decimal / Currency, never float. No PII in logs.
  • Every transfer path validated; ledger entries balanced.
  • Treat each MCP server as new attack surface — least privilege.
  • Schedule Automations deliberately (cost + noise).
  • Keep secrets out of prompts and worktrees.
  • Encode house rules as skills/plugins so review is consistent.
The most important slide for a bank audience. Trainer tips: (1) always demo "show me the diff before applying"; (2) never merge without a human gate; (3) remind that the banking-review skill and security-reviewer agent exist to make review consistent. Tie back to the planted bug: tooling helps, but judgement ships.
